# 汉字笔画拆分接入总体方案

## 一、目标

当前 `moka_wangye` 项目的写字功能在 `app/static/index.html` 中硬编码了 `yongStrokes` 和 `fallbackStrokes`，实际只能较好地生成“永”字轨迹，其他字符只是简单兜底线条。

本方案的目标是接入 `D:\robotcontrol_exp\character_decomposition` 下的汉字笔画数据项目，让网页输入汉字后，可以读取真实笔画中心线数据，生成机械臂可执行的书写轨迹。

核心原则：

- 不把 `makemeahanzi-master` 和 `hanzi-writer-data-master` 直接合并。
- 使用 `makemeahanzi-master` 作为原始数据来源说明和备份。
- 使用 `hanzi-writer-data-master/data/字.json` 作为工程运行时的主要数据源。
- 使用 `medians` 生成机器人笔尖轨迹。
- 使用 `strokes` 做预览或后续可视化增强，第一版不参与运动控制。

## 二、总体数据流

```text
网页输入汉字
  ↓
读取 hanzi-writer-data-master/data/字.json
  ↓
提取 medians
  ↓
坐标缩放、Y 轴修正、排版
  ↓
插入抬笔 / 落笔点
  ↓
生成 [X, Y, Z, RX, RY, RZ] 航点
  ↓
WebSocket: write_path
  ↓
FastAPI 转 WRITE 指令
  ↓
C++ execute_write_path()
  ↓
机械臂执行
```

## 三、两个数据项目的分工

### 1. makemeahanzi-master

定位：原始数据项目。

主要文件：

- `dictionary.txt`：每行一个汉字 JSON，包含释义、拼音、部件结构拆分、部首、笔画到部件的匹配关系。
- `graphics.txt`：每行一个汉字 JSON，包含 `strokes` 和 `medians`。
- `svgs/`：动态笔顺 SVG。
- `svgs-still/`：静态彩色笔顺 SVG。
- `stroke_caps/`：笔画 SVG 修正与生成脚本。

注意：

- `dictionary.txt` 中的 `decomposition` 是部件结构拆分，不是机器人书写轨迹。
- 真正适合生成笔尖路径的是 `graphics.txt` 或 `hanzi-writer-data` 里的 `medians`。

### 2. hanzi-writer-data-master

定位：工程化后的单字 JSON 数据源。

主要文件：

- `data/永.json`、`data/中.json` 等：每个汉字一个 JSON。
- `stroke_data_parser.py`：从 `makemeahanzi` 的 `dictionary.txt` 和 `graphics.txt` 生成单字 JSON。

单字 JSON 主要字段：

```json
{
  "strokes": ["M ...", "M ..."],
  "medians": [
    [[x1, y1], [x2, y2]],
    [[x1, y1], [x2, y2]]
  ],
  "radStrokes": [1, 2]
}
```

本项目优先读取 `medians`。

## 四、现有项目集成点

当前写字功能的主要代码位置：

- `app/static/index.html`
  - `makeWritingPath()`：当前生成书写路径的核心函数。
  - `yongStrokes`：当前硬编码的“永”字笔画。
  - `fallbackStrokes`：其他字符的兜底笔画。
  - `sendWriting()`：将生成的 `points` 通过 WebSocket 发送出去。

- `app/app.py`
  - WebSocket 接收 `write_path`。
  - 将点序列打包成 `WRITE ...` 指令发送给 C++。

- `wangye.cpp`
  - 解析 `WRITE` 指令。
  - `execute_write_path()` 将笛卡尔航点转换为机器人轨迹。

第一阶段应优先替换 `index.html` 中的硬编码笔画生成逻辑。

## 五、关键技术问题

### 1. 坐标系转换

`hanzi-writer-data` 的坐标来自 Make Me a Hanzi，属于 1024 字形坐标系，不是普通屏幕坐标。

需要处理：

- 字形坐标包围盒计算。
- 缩放到用户设置的书写宽高。
- Y 轴方向修正。
- 平移到纸面中心点。
- 转换成机器人工作空间坐标。

推荐做法：

```text
收集当前字符所有 median 点
  ↓
计算 minX / maxX / minY / maxY
  ↓
归一化到 0..1
  ↓
缩放到 write-width / write-height
  ↓
以 write-x / write-y 为中心平移
```

### 2. 抬笔和落笔

每个笔画都应生成：

```text
笔画起点，Z = paperZ + lift
笔画起点，Z = paperZ
笔画中间点，Z = paperZ
笔画终点，Z = paperZ
笔画终点，Z = paperZ + lift
```

这样才能避免不同笔画之间被连线。

### 3. 点数限制

当前 C++ 底层 `execute_write_path()` 中存在 80 点上限：

```cpp
size_t max_points = points.size() > 80 ? 80 : points.size();
```

如果直接把复杂汉字所有 median 点发送给底层，可能会被截断。

第一版建议：

- 前端先限制总点数不超过 80。
- 对复杂字做降采样。
- 预览和实际发送使用同一份点。
- 点数超限时给用户提示。

后续再修改 C++ 队列和协议，支持更多点。

### 4. 轨迹平滑

第一版不建议直接做复杂 B 样条或曲率速度规划。

原因：

- 当前 C++ 已经会做插补。
- 过度平滑可能改变笔画形态。
- 平滑后点数可能增加，触发底层 80 点限制。

第一版只做：

- 去除重复点。
- 去除过近点。
- 简单按距离重采样。

## 六、分阶段执行规划

### 阶段 1：单字数据接入

目标：把“永”从硬编码改成读取 JSON。

任务：

- 在 `app/static/` 下新增本地数据目录，例如 `hanzi-data/`。
- 先复制少量测试字 JSON，例如：
  - `永.json`
  - `中.json`
  - `国.json`
  - `人.json`
  - `机.json`
- 修改 `makeWritingPath()`，根据输入字符读取对应 JSON。
- 从 JSON 中提取 `medians`。
- 保留当前硬编码逻辑作为 fallback。

验收标准：

- 输入“永”能生成和原来接近的预览。
- 输入已复制的数据字能生成真实笔画轨迹。
- 输入不存在的字能提示缺少字库数据。

### 阶段 2：字体坐标到纸面坐标

目标：把 `medians` 坐标稳定映射到机械臂书写区域。

任务：

- 解析所有 median 点，计算单字包围盒。
- 按用户输入的宽、高缩放。
- 修正 Y 轴方向。
- 以“中心 X/Y + 字宽/字高”定位。
- 每笔生成抬笔、落笔和书写点。

验收标准：

- 预览不倒置。
- 字能写在用户设置的中心点附近。
- 每一笔之间不会被错误连线。

### 阶段 3：点数控制与重采样

目标：让轨迹既保持字形，又不会超过当前底层能力。

任务：

- 去除重复点。
- 去除距离过近点。
- 按距离重采样，控制点间距，例如 `3-8mm`。
- 设置总点数上限。
- 第一版保持 80 点以内。
- 对复杂字提示降低精度或减少字数。

验收标准：

- 单字轨迹不会被 C++ 截断。
- 复杂字能降采样后执行。
- 预览点数和实际发送点数一致。

### 阶段 4：后端字库服务

目标：避免前端直接管理大量 JSON 文件。

任务：

- 在 `app.py` 中增加接口，例如：

```text
GET /hanzi/{char}
```

- 服务端从配置的数据目录读取：

```text
D:\robotcontrol_exp\character_decomposition\hanzi-writer-data-master\data\{char}.json
```

- 前端改为请求 `/hanzi/永`。
- 缺字时返回明确错误。

验收标准：

- 不需要把 9000 多个 JSON 全部复制到前端目录。
- 前端请求失败时有明确提示。
- 后续部署只需要配置数据路径。

### 阶段 5：底层路径能力增强

目标：突破 80 点限制，支持更复杂字和多字文本。

任务：

- 修改 `wangye.cpp` 中 `execute_write_path()` 的点数上限。
- 评估轨迹队列长度、实时循环消费速度和内存安全。
- 支持更大队列或分批路径。
- 为 `WRITE` 指令增加解析校验和日志。
- 日志中输出：
  - 接收点数
  - 实际执行点数
  - 是否拒绝
  - 是否截断

验收标准：

- 能执行超过 80 点的单字。
- 多字不会被静默截断。
- 底层日志能明确显示路径状态。

### 阶段 6：多字排版

目标：支持“中国”“机器人”等短文本。

任务：

- 每个字独立读取 `medians`。
- 做横向排版。
- 加入字符间距。
- 支持整体居中。
- 第一版限制最大字数，例如最多 3 个字。
- 每个字之间插入抬笔移动。
- 预览中显示完整文本。

验收标准：

- 多字之间不重叠。
- 文本整体位于书写区域内。
- 超出区域时拒绝发送。

### 阶段 7：运动质量优化

目标：让字写得更稳、更自然。

任务：

- 对每一笔做轻量平滑。
- 根据笔画长度和曲率调整采样密度。
- 对首段跳跃距离、单段距离、Z 轴高度做更严格检查。
- 可选：按笔画设置速度，短笔慢一点，长笔快一点。

验收标准：

- 轨迹不明显抖动。
- 机械臂不会出现大幅跳变。
- 字形仍然接近原始笔顺数据。

## 七、推荐优先级

### 第一轮：MVP

优先完成：

- 阶段 1：单字数据接入。
- 阶段 2：字体坐标到纸面坐标。
- 阶段 3：点数控制与重采样。

目标效果：

```text
输入一个汉字
读取本地 JSON
提取 medians
生成预览
生成 80 点以内机器人航点
通过现有 write_path 链路发送
```

### 第二轮：工程化

完成：

- 阶段 4：后端字库服务。
- 阶段 5：底层路径能力增强。

目标效果：

```text
前端不直接依赖大量 JSON
底层支持更多点
缺字、超限、截断都有明确反馈
```

### 第三轮：体验与效果

完成：

- 阶段 6：多字排版。
- 阶段 7：运动质量优化。

目标效果：

```text
支持短文本书写
字形更稳定
运动更平顺
安全检查更完善
```

## 八、MVP 范围建议

最小可行版本只做：

```text
输入一个汉字
读取 hanzi-writer-data JSON
提取 medians
坐标缩放
加入 Z 轴抬落笔
限制点数
预览
发送给现有 C++ 底层
```

暂时不做：

```text
完整多字排版
复杂 B 样条平滑
毛笔笔触粗细控制
力控
图片文字识别
书法风格迁移
ROS2 接口
```

## 九、结论

该方案技术上可行，并且和现有项目结构匹配度较高。

最稳妥的落地方式是先在前端替换硬编码笔画，把 `hanzi-writer-data` 的 `medians` 转换成现有 `write_path` 所需的 `[X, Y, Z, RX, RY, RZ]` 点列。这样可以最大限度复用当前 WebSocket、FastAPI 和 C++ EtherCAT 控制链路。

第一阶段完成后，项目就能从“只能写硬编码永字”升级为“可读取真实汉字笔画数据并生成机器人航点”的通用单字书写系统。
