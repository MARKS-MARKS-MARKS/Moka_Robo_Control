# 智能答题功能实施方案

## 目标

在当前实体机器人控制项目中新增“看题、理解、回答、书写”的智能交互流程：

1. 机器人移动到预设拍照位姿
2. 末端相机采集纸面问题区域图像
3. 视觉识别或大语言模型 API 读取题目内容
4. 生成简短答案
5. 清洗并限制答案文本
6. 复用现有汉字轨迹生成与 `write_path` 写字链路
7. 机器人在指定答案区域写出回答

## 当前项目可复用能力

### 底层控制

- `wangye.cpp`
  - EtherCAT 实时控制
  - `MOVEL` TCP 空间运动
  - `WRITE` 多航点书写轨迹
  - `GRIPPER` / IO 控制

### Python 后端

- `app/app.py`
  - FastAPI 服务
  - WebSocket 指令转发
  - `/camera/status`
  - `/camera/stream`
  - `/hanzi/{char}`
  - `/teach-path`

### 前端页面

- `app/static/index.html`
  - 机器人连接与状态显示
  - TCP 控制
  - 汉字书写
  - 井字棋书写
  - 示教再现
  - 相机预览

## 阶段一：需求参数固化

目标：先把所有现场参数明确下来，避免一开始就写复杂代码。

### 当前现场默认参数

拍照位姿：

```text
photo_x  = 591.73
photo_y  = -0.50
photo_z  = 918.09
photo_rx = -3.1416
photo_ry = 1.3963
photo_rz = 0
```

答案书写参考位姿：

```text
answer_x  = 450
answer_y  = 0
answer_z  = 549
answer_rx = 3.14
answer_ry = 0
answer_rz = 0
```

说明：

- `photo_*` 用于机器人移动到拍照位置后采集问题区域图像
- `answer_*` 用于答案书写区域的默认中心和姿态
- 后续前端“智能答题”面板应默认填入这些参数

### 需要确定的参数

- 拍照位姿：

```text
photo_x
photo_y
photo_z
photo_rx
photo_ry
photo_rz
```

- 答案书写区域：

```text
answer_center_x
answer_center_y
answer_paper_z
answer_lift
answer_width
answer_height
answer_sample_step
answer_rx
answer_ry
answer_rz
```

- 问题图像范围：
  - 先使用整张相机图像
  - 后续再增加裁剪框

- 答案限制：

```text
max_answer_chars = 3
```

### 验收标准

- 能在页面中手动填入并保存拍照位姿
- 能在页面中手动填入并保存答案区域参数
- 参数刷新页面后不丢失

## 阶段二：相机单帧采集接口

目标：在现有相机预览基础上增加“拍一张图”的能力。

当前状态：已实现。

### 后端建议新增接口

```text
GET /camera/capture
```

返回：

```json
{
  "ok": true,
  "imageBase64": "...",
  "width": 640,
  "height": 360,
  "capturedAt": 1780000000000

```

### 实现要点

- 复用当前 DepthAI + OpenCV 初始化逻辑
- 采集单帧 JPEG
- 返回 base64，便于前端显示和发给模型
- 如果相机被 `/camera/stream` 占用，需要提示先停止预览

### 当前实现

- 后端新增：

```text
GET /camera/capture
```

- 返回字段：

```text
ok
imageBase64
width
height
capturedAt
```

- 前端相机面板新增“拍照”按钮
- 拍照成功后复用相机显示区域展示静态图像
- 若正在预览导致相机占用，会提示先停止预览

### 验收标准

- 点击“拍照”后页面显示静态图片
- 相机不可用时返回明确错误
- 不影响现有 `/camera/stream`

## 阶段三：AI 识别与回答接口

目标：后端接收图像，调用视觉模型或 OCR + 文本模型，返回题目和答案。

当前状态：已实现最小验证版本。

### 推荐接口

```text
POST /qa/answer
```

请求：

```json
{
  "imageBase64": "...",
  "maxAnswerChars": 3
}
```

返回：

```json
{
  "ok": true,
  "question": "一加一等于几",
  "rawAnswer": "答案是二。",
  "cleanAnswer": "二"
}
```

### 模型策略

优先使用视觉大模型直接读图：

```text
图像 -> 视觉模型 -> 题目文本 + 简短答案
```

备选方案：

```text
图像 -> OCR -> 文本模型 -> 简短答案
```

### 当前实现

- 后端新增：

```text
POST /qa/answer
```

- 使用 OpenAI Responses API 视觉输入
- 不额外依赖 `openai` Python 包，使用 Python 标准库 HTTP 请求
- 默认模型：

```text
gpt-4.1-mini
```

- 可通过环境变量覆盖：

```text
QA_VISION_MODEL
OPENAI_MODEL
```

- 必需环境变量：

```text
OPENAI_API_KEY
```

- 智谱 GLM-5V-Turbo 视觉答题配置：

```text
QA_PROVIDER=zhipu
ZHIPU_API_KEY=你的智谱 API Key
ZHIPU_MODEL=glm-5v-turbo
```

- GLM-5V-Turbo 当前作为“图片题目 -> 简短答案”的视觉回答模型使用
- 使用 GLM-5V-Turbo 时，`POST /qa/answer` 可直接传入 `imageBase64`
- 如果已经有 OCR 文本，也可以传入 `questionText` 做纯文本答题
- 当前完整链路调整为：

```text
图像 -> GLM-5V-Turbo -> cleanAnswer -> 书写
```

- 前端相机面板新增“识别答题”按钮
- “识别答题”会使用最近一次“拍照”的图片调用 `/qa/answer`
- 页面会显示模型名称、原始答案和清洗答案

如果未配置 `OPENAI_API_KEY`，接口会返回明确错误，不影响相机、机器人控制和写字功能。

### Prompt 约束建议

```text
请读取图片中的手写或印刷题目，并给出最短答案。
只输出答案，不要解释。
答案限制在 3 个中文字符以内。
如果无法识别，请输出“看不清”。
```

### 验收标准

- 能识别简单算术题
- 能识别常见中文问答题
- 模型失败时返回可显示错误
- 不会返回长篇解释

## 阶段四：答案文本清洗

目标：模型输出不能直接进入书写模块，必须先清洗。

### 清洗规则

- 去掉换行和多余空格
- 去掉“答案是”“所以”“结果为”等前缀
- 去掉大部分标点
- 保留中文、数字、英文字母、简单数学符号
- 限制长度，例如最多 3 个字符
- 空答案时使用：

```text
看不清
```

### 建议函数

```python
def clean_answer_text(raw_answer: str, max_chars: int = 3) -> str:
    ...
```

### 验收标准

- `答案是二。` -> `二`
- `所以结果为 42` -> `42`
- 空字符串 -> `看不清`
- 长句会被截断

## 阶段六：移动到拍照位

目标：复用现有 `movel_cart` 发送拍照位姿。

### 前端发送

```json
{
  "type": "movel_cart",
  "target": [x, y, z, rx, ry, rz]
}
```

### 到位判断

复用示教再现中的状态判停逻辑：

```text
每 2 秒监测一次关节状态
连续两次最大关节变化 <= 0.05deg
认为机器人停止
```

### 验收标准

- 机器人能移动到拍照位
- 到位后再拍照
- 状态链路异常时停止流程

## 阶段七：答案书写轨迹生成

目标：复用当前汉字书写模块，不重新造一套写字逻辑。

### 方案

前端拿到 `cleanAnswer` 后：

1. 自动填入 `write-text`
2. 自动填入答案区域参数
3. 调用现有 `makeWritingPath`
4. 使用现有 `write_path`

### 注意事项

- 答案不能太长
- 轨迹点不能超过 `MAX_WRITE_POINTS`
- 首段距离过大时需要提示先移动到答案区域附近
- 写出前必须确认

### 验收标准

- 简短答案能预览
- 简短答案能写到指定区域
- 超长答案会被拒绝或截断

## 阶段八：一键执行流程

目标：串联完整流程。

### 推荐顺序

```text
1. 检查 WebSocket 已连接
2. 检查相机可用
3. 弹窗确认周围安全
4. MOVEL 到拍照位
5. 等待机器人停止
6. 拍照
7. 调用 AI 识别并回答
8. 清洗答案
9. 生成答案书写轨迹
10. 弹窗确认答案和书写区域
11. 发送 write_path
12. 显示完成状态
```

### 验收标准

- 单击“一键执行”能完成完整闭环
- 任一步失败都会停止后续动作
- 错误信息明确显示在页面上

## 阶段十：安全与异常处理

### 必须处理的异常

- WebSocket 未连接
- 底层控制程序未运行
- 机器人状态长时间不刷新
- 相机不可用
- 拍照失败
- 模型 API 失败
- 模型返回空答案
- 答案太长
- 字库缺字
- 轨迹点超过上限
- 首段距离过大
- 坐标超出安全范围

### 安全策略

- 所有实际运动前弹窗确认
- 所有写字前弹窗确认
- 坐标范围继续使用现有安全范围
- 答案长度默认限制到 3 个字符
- 调试阶段先高 Z 空跑

## 推荐开发顺序

### 第 1 步：只做 UI 和配置

- 新增智能答题面板
- 添加参数输入
- 参数保存到 `localStorage`
- 不调用相机，不调用模型，不运动

### 第 2 步：接入单帧拍照

- 新增 `/camera/capture`
- 页面显示拍照结果
- 验证相机稳定性

### 第 3 步：接入 AI 识别

- 新增 `/qa/answer`
- 先只返回题目和答案
- 不让机器人写字

### 第 4 步：接入答案清洗

- 后端实现 `clean_answer_text`
- 页面展示原始答案和清洗答案

### 第 5 步：接入写字预览

- 将清洗答案填入现有写字模块
- 生成预览
- 不发送机器人

### 第 6 步：接入答案书写

- 使用现有 `write_path`
- 写之前弹窗确认
- 先高 Z 空跑

### 第 7 步：接入拍照位移动

- 使用 `movel_cart`
- 复用状态判停
- 到位后拍照

### 第 8 步：一键执行

- 串联完整流程
- 加入每一步状态提示和错误中断

## 不建议改动的部分

- 不建议改 `general_6s.cpp`
- 不建议把 AI 逻辑写进 `wangye.cpp`
- 不建议新增底层命令
- 不建议绕过现有 `write_path`

最小可行版本只需要：

1. `/camera/capture`
2. `/qa/answer`
3. 前端智能答题面板
4. 复用现有 `makeWritingPath`
5. 复用现有 `write_path`

底层 C++ 不需要改。
